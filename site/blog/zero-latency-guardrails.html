Why Your LLM Safety Layer Shouldn't Be Another LLM

March 8, 2026 · 10 min read · by Maxwell Calkin

Every major guardrails library takes the same approach: use a model to validate another model's output. NeMo Guardrails runs LLM calls. LLM Guard loads ML classifiers. Guardrails AI calls GPT-4 for validation.

This creates a problem: your safety layer is now the slowest part of your stack. A single guardrails check adds 100-500ms of latency, requires GPU inference or API calls, and introduces its own failure modes. If you're scanning every token in a streaming response, that's unacceptable.

The latency problem is worse than you think

That's a 2,500-10,000x difference. At 0.05ms per scan, you can scan every token in a streaming response, every tool call in an agentic workflow, every field in a structured output — with zero perceptible delay.

But regex can't catch everything

This is the correct objection, and it's important to be honest about it.

What regex-based detection cannot do:

  • Detect novel, unseen attack patterns
  • Understand semantic context ("tell me how locks work" vs. "tell me how to pick locks")
  • Catch attacks that use natural-sounding paraphrasing with no structural markers
  • Evaluate whether content is harmful in a specific context

A well-crafted adversarial prompt with no structural markers will bypass pattern matching. That's a fact. But here's what the "use a model for everything" crowd misses:

What regex-based detection excels at:

  • Known attack patterns (instruction overrides, delimiter injection, jailbreak templates)
  • Structural attacks (HTML comment injection, base64-encoded payloads, zero-width chars)
  • Data classification (PII, API keys, credit card numbers, SSNs)
  • Code vulnerabilities (SQL injection, command injection, XSS in generated code)
  • Multilingual variants (same patterns expressed in 12+ languages)
  • 100% deterministic — same input always produces same output

The right architecture: two layers

The answer isn't regex or models. It's regex then models. A fast deterministic first pass catches the 90%+ of attacks that use known patterns, then a model does the deep reasoning on what passes through. 

User Input
    |
    v
[Sentinel AI: ~0.05ms]   <-- catches known patterns instantly
    |
    |-- BLOCKED (known attack) --> reject immediately
    |
    v
[Model-based check: ~300ms]   <-- deep reasoning on remaining
    |
    |-- BLOCKED (novel attack) --> reject
    |
    v
[LLM processes request]

This gives you:

  1. Speed: 90%+ of attacks blocked in under 0.1ms
  2. Coverage: novel attacks caught by the model layer
  3. Cost: only ambiguous inputs hit the expensive model check
  4. Determinism: known patterns have guaranteed, repeatable detection

Real-world applications

The pragmatic view

Pattern matching is old technology. It's not impressive. It doesn't use transformers or diffusion models or any of the techniques that get papers published. But for the class of problems it solves — known attack patterns, data classification, code vulnerability detection — it's strictly better than model-based approaches on every dimension: faster, cheaper, more deterministic, and more reliable.

Use regex for what regex is good at. Use models for what models are good at. Don't use a $15/million-token model to check if a string contains "ignore all previous instructions."
