Turning CLAUDE.md Rules into Deterministic Guardrails

March 8, 2026 · 7 min read

Every Claude Code project has a CLAUDE.md file with rules like "Never use rm -rf" or "Do not access .env." These rules are critical for safety — but they're just suggestions. The model reads them, understands them, and usually follows them. But "usually" isn't good enough when the stakes are production data, credentials, or infrastructure.

Today we're releasing ClaudeMdEnforcer: a tool that reads your CLAUDE.md, extracts the prohibition rules, and converts them into deterministic checks that actually block violations before they execute.

The Problem: Rules Without Enforcement

Consider a typical CLAUDE.md:

# Project Rules
- Never use `rm -rf`
- Do not access `.env` or `.aws/credentials`
- Never run `git push --force`
- Do not modify files in `tests/` without asking

These rules exist in the model's context window. The model sees them and generally complies. But there are failure modes:

When CLAUDE.md rules fail:
  • Context window overflow — Long sessions push CLAUDE.md instructions out of the context window
  • Prompt injection — An attacker's payload in a file overrides CLAUDE.md instructions
  • Jailbreaks — Adversarial prompts convince the model to ignore its instructions
  • Model updates — A new model version may interpret rules differently
  • Ambiguity — "Do not modify tests/" — does renaming a test file count?

The fundamental issue: CLAUDE.md rules are enforced by the same system they're trying to constrain. That's like asking a program to enforce its own security policy without an OS kernel.

The Solution: Deterministic Enforcement

ClaudeMdEnforcer adds a kernel-level enforcement layer. It parses your CLAUDE.md, extracts prohibition rules using NLP pattern matching, and generates deterministic checks that run outside the model's control:

from sentinel import ClaudeMdEnforcer

# Parse your CLAUDE.md
enforcer = ClaudeMdEnforcer.from_file("CLAUDE.md")

# Check a tool call
verdict = enforcer.check("bash", {"command": "rm -rf /tmp"})
print(verdict.allowed)       # False
print(verdict.violated_rules) # ["Never use `rm -rf`"]

# See what rules were extracted
print(enforcer.summary())
# Extracted 4 enforceable rule(s) from CLAUDE.md:
#   Blocked Commands (2):
#     L2 rm -rf  — Never use `rm -rf`
#     L4 git push --force  — Never run `git push --force`
#   Blocked Paths (2):
#     L3 .env  — Do not access `.env` or `.aws/credentials`
#     L5 tests/  — Do not modify files in `tests/`

How It Works

ClaudeMdEnforcer uses a three-stage pipeline:

1. Prohibition Detection

The parser identifies prohibition rules using linguistic patterns:

PatternExample
never X"Never use rm -rf"
do not X"Do not access .env"
must not X"You must not run wget"
X is forbidden"curl is forbidden"
avoid X"Avoid modifying the database"

It handles Markdown formatting: list items, backtick-quoted content, bold/italic, and prose paragraphs. Regular documentation (like "the user should not see errors") is ignored — only actionable tool-constraining rules are extracted.

2. Command & Path Extraction

Once a prohibition is identified, the parser extracts the specific command, path, or tool being blocked:

  • "Never use `rm -rf`" → blocked_command: rm -rf
  • "Do not modify files in `tests/`" → blocked_path: tests/
  • "curl is forbidden" → blocked_tool: curl
  • "Never run `git push --force`" → blocked_command: git push --force

3. Runtime Enforcement

Each extracted rule becomes a compiled regex that's tested against every tool call. The check runs in microseconds — no LLM inference, no API calls, no latency.

Integration with Claude Code Hooks

The killer feature: ClaudeMdEnforcer plugs directly into Claude Code's PreToolUse hook system. Every tool call is checked before execution:

# Install sentinel and configure hooks:
pip install sentinel-guardrails
sentinel init

# Or manually add to .claude/settings.json:
{
  "hooks": {
    "PreToolUse": [{
      "matcher": ".*",
      "hooks": [{
        "type": "command",
        "command": "sentinel hook"
      }]
    }]
  }
}

Now every Bash, Write, and Edit call is checked against your CLAUDE.md rules. If a rule is violated, the tool call is blocked with exit code 2 and a clear message explaining why.

CLI Usage

The sentinel enforce command gives you full access from the terminal:

# Extract and display rules
sentinel enforce --file CLAUDE.md --show-rules

# Check a specific tool call
sentinel enforce --file CLAUDE.md --tool bash --command "rm -rf /tmp"
# Output: [BLOCKED] bash — Rule violated: Never use `rm -rf`

# Export as a guard policy YAML
sentinel enforce --file CLAUDE.md --export-policy > policy.yaml

# JSON output for CI/CD integration
sentinel enforce --file CLAUDE.md --export-policy --format json

Policy Generation

ClaudeMdEnforcer can export its extracted rules as a GuardPolicy — a declarative policy object that works with SessionGuard for full session-level enforcement:

enforcer = ClaudeMdEnforcer.from_file("CLAUDE.md")

# Convert to a guard policy
policy = enforcer.to_guard_policy()

# Create a session guard with the policy
guard = policy.create_guard(session_id="agent-session-1")

# Now every tool call is checked
verdict = guard.check("bash", {"command": "rm -rf /"})
# Blocked by CLAUDE.md rule AND by built-in dangerous command detection

This gives you layered defense: CLAUDE.md rules plus built-in safety scanning, all in one check() call.

TypeScript SDK

ClaudeMdEnforcer is also available in the TypeScript SDK:

What Gets Extracted (and What Doesn't)

Extracted as enforceable rules:
  • "Never use rm -rf" → blocks any bash command containing rm -rf
  • "Do not access .env" → blocks file reads/writes to .env
  • "curl is forbidden" → blocks all curl tool calls
  • "Blocked commands: mkfs, dd, fdisk" → blocks each listed command
Not extracted (by design):
  • "The application should handle errors gracefully" — not a tool constraint
  • "Use TypeScript for new files" — affirmative preference, not a prohibition
  • "Follow PEP 8 style" — style guidance, not a security rule

The parser deliberately focuses on security-relevant prohibitions — rules that constrain what tools can do. Style preferences and architectural guidance remain in the model's context where they belong.

Defense in Depth

ClaudeMdEnforcer is one layer in a defense-in-depth strategy:

LayerWhatEnforced By
1CLAUDE.md rulesModel context (soft)
2ClaudeMdEnforcerDeterministic regex (hard)
3SessionGuardAttack chain detection (smart)
4GuardPolicyDeclarative YAML policy (configurable)

Layer 1 handles the 99% case. Layers 2-4 catch the 1% that matters most.
