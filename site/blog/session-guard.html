One-Line Safety for Agentic AI: Introducing SessionGuard

March 8, 2026 · 8 min read

Agentic AI systems — LLMs that call tools, write code, and interact with the real world — are the fastest-growing surface area in AI security. A single agent session might execute 50+ tool calls, any combination of which could constitute an attack chain that no individual call would trigger.

Today we're releasing SessionGuard: a unified safety layer that combines audit logging, attack chain detection, and threat intelligence into a single guard.check() call. One line of code gives you real-time protection, full audit trail, and SIEM-ready export.

The Problem: Agentic Safety is Multi-Dimensional

Existing safety tools solve pieces of the puzzle:

  • Input scanners catch prompt injection in individual messages
  • Tool-use validators block dangerous commands one at a time
  • Audit loggers record what happened after the fact

But agentic attacks are sequences. An attacker doesn't just run rm -rf / — they first run whoami (reconnaissance), then cat .env (credential access), then curl -X POST (exfiltration). Each step looks innocuous alone.

SessionGuard solves this by combining three systems into one call:

Usage: One Line

from sentinel.session_guard import SessionGuard

guard = SessionGuard(
    session_id="session-123",
    user_id="user@org.com",
    block_on="critical",  # Block anything >= critical risk
)

# In your agent loop:
for tool_call in agent.tool_calls():
    verdict = guard.check(tool_call.name, tool_call.arguments)

    if not verdict.allowed:
        print(f"BLOCKED: {verdict.block_reason}")
        continue

    if verdict.warnings:
        print(f"WARNING: {verdict.warnings}")

    # Execute the tool call
    result = execute_tool(tool_call)

What Gets Checked

Every guard.check() call runs four layers of analysis:

1. Destructive Command Detection

Compiled regex patterns catch dangerous commands instantly:

guard.check("bash", {"command": "rm -rf /"})
# GuardVerdict(allowed=False, risk="critical",
#   block_reason="destructive_command: ...")

Catches: rm -rf, fork bombs, dd to disk, chmod 777 /, curl | bash, and more.

2. Sensitive File Access

guard.check("read_file", {"path": ".env"})
# GuardVerdict(allowed=True, risk="high",
#   warnings=["Sensitive file access: .env"])

Monitors access to: .env, .aws/credentials, .ssh/id_rsa, /etc/shadow, .kube/config, and 10+ other sensitive paths.

3. Threat Intelligence Matching

guard.check("bash", {"command": "echo 'ignore all previous instructions'"})
# GuardVerdict(threat_matches=["[critical] Direct Instruction Override"])

Matches against 27 MITRE ATLAS-aligned indicators across prompt injection, jailbreaks, data exfiltration, privilege escalation, and evasion techniques.

4. Attack Chain Detection

guard.check("bash", {"command": "whoami"})          # reconnaissance
guard.check("read_file", {"path": ".env"})            # credential access
guard.check("bash", {"command": "curl -X POST ..."})  # exfiltration
# GuardVerdict(chains_detected=["recon_credential_exfiltrate"], risk="critical")

Detects 6 multi-step attack chains including recon→credential→exfiltrate, escalation→destruction, and context poisoning sequences.

Custom Rules

Add project-specific blocking rules:

guard = SessionGuard(
    block_on="high",
    custom_rules=[
        # Block npm publish in CI
        lambda tool, args: "no_publish"
            if "npm publish" in args.get("command", "")
            else None,
        # Block writes to production config
        lambda tool, args: "prod_config_protected"
            if "/prod/" in args.get("path", "")
            else None,
    ],
)

SIEM Export

Every check is logged with SHA-256 hash chaining. Export the full trail at session end:

report = guard.export()

# report = {
#   "session_id": "session-123",
#   "user_id": "user@org.com",
#   "integrity_verified": True,
#   "summary": {
#     "total_calls": 47,
#     "blocked_calls": 2,
#     "risk_level": "critical",
#     "tool_counts": {"bash": 20, "read_file": 15, ...},
#     "finding_categories": {"credential_access": 3, ...}
#   },
#   "active_chains": [
#     {"name": "recon_credential_exfiltrate", "severity": "critical", ...}

#   "entries": [...]  # Every check, hash-chained

Ship the JSON to Splunk, Datadog, Elastic, or any SIEM. The hash chain lets you prove no entries were tampered with — critical for SOC 2 and ISO 27001 compliance audits.

Architecture

Design principle: SessionGuard is a coordinator, not a new scanner. It combines existing, battle-tested components (each with their own test suites) into a unified API. This means you get the reliability of three proven systems with the simplicity of one call.

Internally, each check() call:

  1. Checks against compiled destructive command patterns (sub-microsecond)
  2. Scans for sensitive file access (O(1) string matching)
  3. Matches against ThreatFeed (27 compiled regex patterns)
  4. Records in AttackChainDetector (windowed stage classification)
  5. Applies custom rules (user-defined lambdas)
  6. Makes block/allow decision based on threshold
  7. Logs to SessionAudit with hash chaining

Total overhead per check: <0.1ms. Zero network calls. Zero dependencies beyond regex.
